# Demo Projects

The `Demo/` folder contains a set of MSBuild projects that demonstrate the complete CQL-to-measure pipeline: from CQL source files through Java-based ELM generation, through the .NET PackagerCLI, and on to packaged FHIR `Library` resources and runnable .NET assemblies.

This document describes the Demo project structure, the pipeline stages, and how to configure a measure project. For details on the build scripts, MSBuild props/targets files, and Java tooling, see [CQL Build Pipeline](build-pipeline.md).

## Prerequisites

Before building the Demo projects you need:

| Tool | Minimum version | Notes |
|------|-----------------|-------|
| .NET SDK | 8.0 or 10.0 | SDK is multi-targeted |
| JDK | 21+ | Required only when `CqlToolingEnabled=true` |
| [Apache Maven](https://maven.apache.org/install.html) | 3.9+ | Downloads CQL-to-ELM CLI JARs from Maven Central |
| PowerShell Core (`pwsh`) | 7+ | Required on Windows; Bash equivalents cover Linux/macOS |

> **Note:** JDK and Maven are only required if you want to regenerate ELM from CQL source. The repository already contains pre-generated ELM files, so a plain `dotnet build` works without Java.

## Demo Project Structure

```
Demo/
├── cql-demo.props              # Shared MSBuild properties for all Demo projects
├── CLI/                        # Demo entry-point console app (loads FHIR resources, runs CQL)
├── Cql/                        # Library set project: CQL → ELM (Java-based)
│   └── Build/
│       ├── CqlToElm.Targets.xml          # MSBuild target: invoke Java CQL-to-ELM CLI
│       ├── ElmToCSharp.Targets.xml       # MSBuild target: invoke PackagerCLI (ELM → C# + FHIR)
│       ├── pom.xml                        # Maven descriptor: declares CQL-to-ELM JAR dependencies
│       ├── Java-Dependencies-Vars.ps1/sh  # Version constants (CQL CLI version, paths)
│       ├── Java-Dependencies-Download.ps1/sh # Downloads JARs via Maven (mutex-protected)
│       └── Java-Dependencies-Clean.ps1/sh    # Removes downloaded JARs
├── Elm/                        # Legacy project referencing ELM Targets
├── Measures/                   # Older measures project
├── Measures.Authoring/         # Example of authoring CQL measures with local CQL/Elm directories
├── Measures.Demo/              # Demo measures using the LibrarySets/Demo library set
├── Measures.dqm-content-qicore-2025/  # CMS DQM Content QICore 2025 measures
└── Test.Measures.Demo/         # Integration tests for the demo measures
```

Source CQL and pre-generated ELM files live outside the Demo projects, under `LibrarySets/`:

```
LibrarySets/
├── Demo/
│   ├── Cql/      # CQL source files
│   ├── Elm/      # Pre-generated ELM JSON files (checked in)
│   └── Resources/  # FHIR Library JSON resources (generated by PackagerCLI)
└── dqm-content-qicore-2025/
    ├── Cql/
    └── Elm/
```

## The Pipeline

The Demo projects implement a three-stage pipeline, each stage optionally gated by an MSBuild property:

```
CQL source files
      │  (CqlToolingEnabled=true → Java CQL-to-ELM CLI)
      ▼
ELM JSON files
      │  (ElmToolingEnabled=true → PackagerCLI)
      ▼
C# source files + FHIR Library resources
      │  (standard dotnet build → Roslyn)
      ▼
.NET assembly (embedded in FHIR Library binary attachment)
```

Stages 2 and 3 are always active: the C# files and FHIR resources are committed to the repository, and the assembly is built by the normal MSBuild compilation step. Stage 1 (CQL → ELM) requires Java and Maven, and stage 2 (ELM → C# + FHIR) requires the PackagerCLI binary.

## Build Scripts and Tooling

The pipeline is driven by two cross-platform build scripts at the repository root (`build.ps1` / `build.sh`) and a set of MSBuild props/targets files. Full details — script parameters, examples, two-phase build behaviour, all target file properties, and the Maven `pom.xml` — are documented in [CQL Build Pipeline](build-pipeline.md).

Quick reference:

```bash
# Build with no code generation (uses committed sources)
./build.sh --framework net10.0 --configuration Debug

# Full pipeline: CQL → ELM (Java) → C# + FHIR (PackagerCLI) → assembly
./build.sh --framework net10.0 --enable-cql-to-elm --enable-elm-to-csharp
```

## Configuring a Measure Project

Each measure project (e.g. `Measures.Demo`) follows the same pattern:

```xml
<!-- 1. Import shared properties FIRST -->
<Import Project="..\..\packaging-tooling.props" />

<!-- 2. Declare library-set-specific paths -->
<PropertyGroup>
  <LibrarySet>Demo</LibrarySet>
  <LibrarySetRoot>$(CqlSolutionDir)LibrarySets/$(LibrarySet)</LibrarySetRoot>
  <CqlDirectory>$(LibrarySetRoot)/Cql</CqlDirectory>
  <ElmDirectory>$(LibrarySetRoot)/Elm</ElmDirectory>
  <CSharpDirectory>$(MSBuildProjectDirectory)/CSharp</CSharpDirectory>
  <ResourcesDirectory>$(LibrarySetRoot)/Resources</ResourcesDirectory>
</PropertyGroup>

<!-- 3. Import tooling targets LAST -->
<Import Project="..\..\packaging-tooling.Targets.xml" />
```

The `PackagerCLI` project reference ensures the packager binary is built before any `GenerateCSharp` target fires:

```xml
<ProjectReference Include="..\..\Cql\PackagerCLI\PackagerCLI.csproj" />
```

## Running the Demo CLI

After building, run the `Demo/CLI` console app to load FHIR Library resources, execute CQL, and print results:

```bash
dotnet run --project Demo/CLI -- --fhir LibrarySets/Demo/Resources
```

The CLI project (`Demo/CLI/CLI.csproj`) references `Hl7.Cql.Invocation` and `Hl7.Cql.Packaging` to load the pre-packaged FHIR resources and invoke the measure logic through the Invocation Toolkit.
