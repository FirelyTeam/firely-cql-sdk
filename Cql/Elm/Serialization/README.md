## Overview

The ELM JSON serialization system handles the conversion of `Library` objects to and from JSON format. This is more complex than standard JSON serialization due to:

1. **Polymorphic type hierarchies** - ELM has deep inheritance trees (e.g., `Element` → `Expression` → `OperatorExpression` → specific operators)
2. **Type discriminator conflicts** - ELM uses a `"type"` property as a discriminator, but some ELM types also have actual properties named `"type"`
3. **Legacy format support** - Must handle JSON generated by older CQL-to-ELM translators
4. **XmlInclude-based type discovery** - Type hierarchies are defined via `[XmlInclude]` attributes

### 2. PolymorphicObjectJsonConverter\<T\>

Handles polymorphic serialization for object types with `"type"` discriminators.

```

│              PolymorphicObjectJsonConverter<T>                  │
├─────────────────────────────────────────────────────────────────┤
```

```
         ↓

         ↓
         ↓
JsonSerializer.Serialize(container, options)
         ↓
         └─→ JsonSerializerOptions configured with:
         │     • PolymorphicObjectJsonConverter<Element>
         │     • PolymorphicObjectJsonConverter<Expression>
         │     • PolymorphicObjectJsonConverter<TypeSpecifier>
         │     • PolymorphicObjectJsonConverter<ExpressionDef>
         │     • PolymorphicObjectJsonConverter<CqlToElmBase>
         │     • PolymorphicObjectJsonConverter<RelationshipClause>
         │     • PolymorphicObjectJsonConverter<AliasedQuerySource>
         │     • PolymorphicObjectJsonConverter<SortByItem>
         │     • TopLevelDefinitionConverterFactory
         │     • XmlQualifiedNameConverter
         │     • JsonStringEnumConverter
         │     • PolymorphicTypeResolver (TypeInfoResolver)
         ↓
         ↓
For each polymorphic property:
         ↓
         └─→ PolymorphicObjectJsonConverter<T>.Write()
         │     ↓
         │     └─→ Determine if "type" discriminator needed
         │     │     • actualType != declaredType
         │     │     • declaredType.IsAbstract

         │     ↓
         │     └─→ Get type metadata via modifiers-only resolver
         │     ↓
         │     └─→ For each property:
         │           • Check ShouldSerialize
         │           • Serialize with original options (for nested polymorphism)
         ↓
         ↓
Return JSON string
```

```
         ↓
```

## The "type" Discriminator Problem

### The Core Issue

ELM JSON uses `"type"` as a discriminator property to indicate the concrete type:

```json
{
  "type": "Add",
  "operand": [
    { "type": "Literal", "valueType": "{urn:hl7-org:elm-types:r1}Integer", "value": "1" },
    { "type": "Literal", "valueType": "{urn:hl7-org:elm-types:r1}Integer", "value": "2" }
  ]
}
```

However, some ELM types (like `Retrieve`, `Search`) also have a **property** named `type`:

```json
{
  "type": "Retrieve",
  "dataType": "{http://hl7.org/fhir}Observation",
  "type": { ... }  // This is a property, not a discriminator!
}
```

### System.Text.Json Limitation

System.Text.Json's built-in polymorphism support (`JsonPolymorphismOptions`) cannot distinguish between:
- `"type"` as a discriminator (string indicating the concrete type)
- `"type"` as an actual property (could be any JSON value)

### The Solution

```
┌──────────────────────────────────────────────────────────────────┐
│                    Problem: "type" Ambiguity                     │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  JSON Input:                                                     │
│  {                                                               │
│    "type": "Retrieve",           ← Discriminator (string)        │
│    "dataType": "...",                                            │
│    "type": { ... }               ← Property (object)             │
│  }                                                               │
│                                                                  │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Solution: CorrectLegacyConstructs() pre-processing              │
│                                                                  │
│  1. If "type" is a string → keep as discriminator                │
│  2. If "type" is an object → remove (handled by resultType*)     │
│  3. If "type" is an array → rename to "choice"                   │
│                                                                  │
│  After processing:                                               │
│  {                                                               │
│    "type": "Retrieve",           ← Only discriminator remains    │
│    "dataType": "..."                                             │
│  }                                                               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### Why Custom Converters Are Needed

Even with pre-processing, we need custom converters because:

1. **Transitive conflicts**: A type may not have a `type` property, but its nested properties might
2. **Infinite recursion prevention**: Standard polymorphism + custom resolver can cause infinite loops
3. **Precise control**: We need to remove `"type"` before deserializing to avoid unmapped member errors

### Properties Named "type"

Some ELM types have actual properties named `"type"` (e.g., `TupleElementDefinition.type`, `ChoiceTypeSpecifier.type`). These properties have complex types (objects/arrays) that would conflict with the string `"type"` discriminator used for polymorphism.

**Solution**: The XSD-to-C# generator automatically adds `[System.Text.Json.Serialization.JsonIgnore]` to properties named `"type"` that have complex types. This prevents them from being serialized in JSON while preserving them for XML serialization (which doesn't use type discriminators).

```csharp
// Generated code example:
[System.Xml.Serialization.XmlElementAttribute("type")]
[System.Text.Json.Serialization.JsonIgnore]  // Auto-generated
public TypeSpecifier[] type { get; set; }
```

This works because:
- **XML serialization**: Uses the property normally (no discriminator conflicts)
- **JSON serialization**: Ignores the property (discriminator takes precedence)
- The type information is still available through `resultTypeSpecifier` or other means

```
┌──────────────────────────────────────────────────────────────────┐
│           Types Handled by Custom Converters                     │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  PolymorphicObjectJsonConverter<T> registered for:               │
│                                                                  │
│  • CqlToElmBase      - Annotations with nested type conflicts    │
│  • ExpressionDef     - Can be FunctionDef with complex body      │
│  • Expression        - Many subtypes, some with "type" props     │
│  • TypeSpecifier     - ChoiceTypeSpecifier had "type" array      │
```

## Component Details

### Type Resolution via XmlIncludeAttribute

The serialization system discovers type hierarchies using `[XmlInclude]` attributes:

```csharp
[XmlInclude(typeof(Add))]
[XmlInclude(typeof(Subtract))]
[XmlInclude(typeof(Multiply))]
public abstract class BinaryExpression : OperatorExpression { }
```

This allows the serializer to:
1. Find all possible concrete types for a polymorphic property
2. Map `"type": "Add"` to the `Add` class
3. Map `Add` instances back to `"type": "Add"`

### Modifiers Pipeline

The serialization uses several "modifiers" that customize JsonTypeInfo:

```
┌──────────────────────────────────────────────────────────────────┐
│                      Modifier Pipeline                           │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. ModifyNarrative                                              │
│     └─→ Renames Narrative.Text property to "value" in JSON       │
│                                                                  │
│  2. DoNotSerializeDefaultValues                                  │
│     └─→ Skips properties with default/null values                │
│     └─→ Exception: Interval type always serializes all props     │
│                                                                  │
│  3. HandleSpecifiedProperties                                    │
│     └─→ Handles xxxSpecified pattern from XML serialization      │
│     └─→ Only serializes property if xxxSpecified is true         │
│     └─→ Sets xxxSpecified = true on deserialization              │
│                                                                  │
│  4. RespectXmlIgnoreAttribute                                    │
│     └─→ Properties with [XmlIgnore] are also ignored in JSON     │
│     └─→ Avoids needing duplicate [JsonIgnore] attributes         │
│     └─→ Used for computed properties in partial class extensions │
│                                                                  │
│  5. AllowOldStyleTypeDiscriminators (deserialization only)       │
│     └─→ Adds "type" property handler for non-polymorphic types   │
│     └─→ Validates discriminator matches expected type name       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## Configuration Options

### JsonSerializerOptions Configuration

```csharp
var options = new JsonSerializerOptions
{
    MaxDepth = 4096,
    UnmappedMemberHandling = JsonUnmappedMemberHandling.Disallow,
};

// Custom converters for polymorphic types
options.Converters.Add(new PolymorphicObjectJsonConverter<CqlToElmBase>(allowOldStyle));
options.Converters.Add(new PolymorphicObjectJsonConverter<ExpressionDef>(allowOldStyle));
options.Converters.Add(new PolymorphicObjectJsonConverter<Expression>(allowOldStyle));
options.Converters.Add(new PolymorphicObjectJsonConverter<TypeSpecifier>(allowOldStyle));
options.Converters.Add(new PolymorphicObjectJsonConverter<Element>(allowOldStyle));
options.Converters.Add(new PolymorphicObjectJsonConverter<RelationshipClause>(allowOldStyle));
options.Converters.Add(new PolymorphicObjectJsonConverter<AliasedQuerySource>(allowOldStyle));
options.Converters.Add(new PolymorphicObjectJsonConverter<SortByItem>(allowOldStyle));

// Factory for top-level definition arrays
options.Converters.Add(new TopLevelDefinitionConverterFactory(allowOldStyle));

// Other converters
options.Converters.Add(new XmlQualifiedNameConverter());
options.Converters.Add(new JsonStringEnumConverter());

// Type resolver with modifiers
options.TypeInfoResolver = new PolymorphicTypeResolver(allowOldStyle)
    .WithAddedModifier(ModifyNarrative)
    .WithAddedModifier(DoNotSerializeDefaultValues)
    .WithAddedModifier(HandleSpecifiedProperties);
```

### Serialization vs Deserialization Options

| Feature | Serialization | Deserialization |
|---------|--------------|-----------------|
| `allowOldStyleTypeDiscriminators` | `false` | `true` |
| `emitConcreteBaseTypeDiscriminator` | `false` | `true` |
| `AllowOldStyleTypeDiscriminators` modifier | No | Yes |
| `CorrectLegacyConstructs` pre-processing | No | Yes |

## Legacy Format Support

### Old-Style Type Annotations

Older CQL-to-ELM translators produced type annotations on wrapper objects:

```json
{
  "statements": {
    "type": "Library$Statements",
    "def": [
      { "type": "ExpressionDef", ... }
    ]
  }
}
```

The `PolymorphicArrayJsonConverter` handles this by:
1. Checking for `"type"` property starting with `"Library$"`
2. Skipping it and proceeding to the `"def"` property

### ChoiceTypeSpecifier Legacy Format

Old format used `"type"` as an array:

```json
{
  "type": "ChoiceTypeSpecifier",
  "type": [
    { "type": "NamedTypeSpecifier", "name": "..." },
    { "type": "NamedTypeSpecifier", "name": "..." }
  ]
}
```

`CorrectLegacyConstructs` converts this to:

```json
{
  "type": "ChoiceTypeSpecifier",
  "choice": [
    { "type": "NamedTypeSpecifier", "name": "..." },
    { "type": "NamedTypeSpecifier", "name": "..." }
  ]
}
```

### Type as Object (resultTypeSpecifier)

Some old formats used `"type"` as an object for type information:

```json
{
  "type": "Literal",
  "type": { "type": "NamedTypeSpecifier", "name": "..." }
}
```

`CorrectLegacyConstructs` removes the object-valued `"type"` property, as this information is typically also available in `resultTypeSpecifier` or `resultTypeName`.

## Debug Assertions

The `EnableDebugAssertions` flag controls debug-time validation:

```csharp
LibraryJsonSerializer.EnableDebugAssertions = true; // default

// When enabled, triggers Debug.Fail() if:
// - A property has a value but its xxxSpecified flag is false
// - This catches inconsistencies between property values and their Specified flags
```

This is automatically disabled in the `CqlToolkit` to avoid interrupting execution during bulk processing.
